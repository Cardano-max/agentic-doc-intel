the problem

The legacy pipeline was three services: an OCR model read pdfs to markdown, a regex parser with hardcoded patterns mapped that to a schema, then it stored the result. The regex parser broke on every new vendor layout. It managed 40% header accuracy on the 4 of 15 documents that made it through, and timed out on the rest. A 73% failure rate on the larger files.

the approach

Skip OCR. Render the pdf pages to images and send them to the model with forced tool use, so it has to return json matching the schema. One call. No markdown, no regex, no parsing.

PDF→ page images
150 DPI→ Haiku 4.5
forced tool use→ validation
5 rules, <10ms→ decision
6 outcomes→ structured JSON

If validation fails, it re-extracts with Claude Sonnet 4 using the validation errors as hints (max 2 retries), then routes to a human queue. toolChoice forced to the extract tool is what makes the output reliable.
